# Roadmap

## 1. 建设路线

建议按 6 个阶段推进，其中 `Phase 0` 用于设计冻结和契约对齐，而不是一次把所有模块都做完。

## 2. Phase 0: 文档与契约冻结

目标：

- 完成总体架构和服务边界设计
- 冻结事件模型和 Topic 规划
- 冻结会话状态机
- 冻结外部协议

交付物：

- 架构设计文档
- 事件契约文档
- 服务拆分清单
- API / Schema 草案
- `docs/contracts.md`
- `api/protobuf/realtime_speech.proto`
- `api/json-schema/*.v1.json`

退出标准：

- 团队对服务边界达成一致
- 不再继续直接维护 HTML 导出页作为主文档
- WebSocket 协议和第一批事件契约完成评审并冻结

## 3. Phase 1: 最小可运行链路

目标：

- 跑通实时字幕闭环

范围：

- `speech-gateway`
- `session-orchestrator`
- `Kafka`
- `asr-worker`
- `translation-worker`
- `Redis`

交付物：

- 单租户开发环境
- 可连通的 WebSocket 接口
- 最小端到端链路 demo
- 基础监控指标

退出标准：

- 能稳定完成“音频上行 -> ASR -> 翻译 -> 字幕下发”
- WebSocket 连接成功率 `>= 99.9%`（30 分钟压测窗口）
- 实时字幕端到端延迟 `P95 <= 1500ms` 且 `P99 <= 2500ms`
- ASR partial 产出间隔 `P95 <= 500ms`
- 关键 Topic（`audio.ingress.raw` / `asr.final` / `translation.result`）消费者延迟可在突刺后 5 分钟内回落到基线
- DLQ 比例 `< 0.1%`

## 4. Phase 2: 生产治理补齐

目标：

- 从“能跑”升级到“能扛”

范围：

- 幂等
- 重试
- DLQ
- 限流
- 熔断
- 背压
- 灰度开关

退出标准：

- 常见故障不会直接拖垮整条链路
- 关键异常可以通过监控快速定位

## 5. Phase 3: TTS 与分发体系

目标：

- 支持语音回放和大规模下发

范围：

- `tts-orchestrator`
- TTS 引擎接入
- 文本归一化
- 缓存键策略
- 对象存储
- CDN 分发

退出标准：

- 热点文本具备高缓存命中率
- 音频分发能力从推理链路中解耦

## 6. Phase 4: 控制面与多租户

目标：

- 把系统做成平台，而不是单业务项目

范围：

- `control-plane`
- 租户管理
- 配额管理
- 模型版本管理
- 语言对策略
- 动态路由与灰度

退出标准：

- 多租户之间策略隔离清晰
- 模型与配置可在线切换和回滚

## 7. Phase 5: 容量、压测与弹性

目标：

- 验证平台的上限和弹性能力

范围：

- 长连接压测
- Kafka 容量规划
- GPU 队列调优
- K8s HPA / KEDA
- 故障演练

退出标准：

- 清楚知道系统瓶颈在哪里
- 清楚知道扩容是否真正有效

## 8. 建议优先级

如果资源有限，建议优先顺序如下：

1. 事件契约
2. 会话状态机
3. 网关与编排层
4. ASR 闭环
5. 翻译闭环
6. 可观测性
7. TTS 与分发
8. 控制面

## 9. 主要风险

### 风险一：过早写代码，过晚冻结契约

后果：

- Topic 和消息体反复重构
- 服务之间互相打补丁

### 风险二：网关承载过多业务逻辑

后果：

- 长连接服务不可扩展
- 实例重启影响面过大

### 风险三：只看功能，不做压测与观测

后果：

- 线上瓶颈定位成本极高
- 无法判断扩容是否有效

### 风险四：把 TTS 当成简单模型调用

后果：

- 高并发下成本失控
- 重复合成与缓存击穿严重

## 10. 推荐仓库结构演进

当前目录适合先作为“资料收敛仓库”。

建议演进顺序：

1. 当前阶段保留 HTML 原文，新增 Markdown 设计文档
2. 下一阶段新增 `api/` 和 `services/` 工程骨架
3. 再下一阶段新增 `deploy/`、`tools/`、压测与监控配置

- 保留原始 HTML，作为参考资料，不再作为主维护对象
- 后续新增内容优先写入 Markdown 文档
- 当代码仓库建立后，把 HTML 移入 `docs/archive/` 或 `docs/sources/`
